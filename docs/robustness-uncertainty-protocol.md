# Robustness And Uncertainty Protocol

## Purpose

Milestone 3 evaluates whether the existing forecast-error patterns are stable enough
to use as research diagnostics.

This protocol does not add model families, tune hyperparameters, or create trading
interpretations. It adds uncertainty and ranking checks around the existing baseline
set.

## Research Questions

1. Are model rankings stable across rolling validation windows?
2. Are pairwise forecast-error differences large relative to their resampled
   uncertainty?
3. Do conclusions change across horizon, ticker, metric, or time period?

## Inputs

Diagnostics should be built from validation-slice forecast errors, not from
aggregate report tables alone.

Required columns for the forecast-error panel:

| Column | Meaning |
| --- | --- |
| `date` | Validation date for the target observation |
| `horizon` | `next_day` or `next_week` |
| `ticker` | Market symbol |
| `model` | Forecast model name |
| `target` | Realized-volatility target |
| `forecast` | Model forecast |
| `error` | `forecast - target` |
| `abs_error` | Absolute forecast error |
| `squared_error` | Squared forecast error |

The panel should be generated by the existing evaluation script or a small
companion CLI. Generated CSV outputs belong under `artifacts/reports/` and should
be reproducible from tracked code and cached/public data.

## Rolling-Window Ranking Diagnostic

The rolling-window diagnostic measures whether model rankings persist through
time instead of appearing only in the full validation slice.

Default settings:

- Window length: 252 trading days.
- Step size: 21 trading days.
- Minimum observations per `(horizon, ticker, model, window)`: 126.
- Metrics: MAE and RMSE.
- Ranking direction: lower error is better.

Output rows should include:

| Column | Meaning |
| --- | --- |
| `horizon` | Forecast horizon |
| `ticker` | Market symbol |
| `metric` | `mae` or `rmse` |
| `window_start` | First date included in the rolling window |
| `window_end` | Last date included in the rolling window |
| `model` | Forecast model |
| `observations` | Non-null observations used |
| `value` | Window metric value |
| `rank` | Rank within `(horizon, ticker, metric, window)` |

The tracked Markdown report should summarize:

- best-model counts by horizon and metric;
- average rank by horizon, ticker, and metric;
- instability flags where the best model changes across windows;
- a short interpretation that avoids model-superiority language.

## Block-Bootstrap Error-Difference Diagnostic

The block-bootstrap diagnostic estimates uncertainty for pairwise model error
differences while preserving short-run dependence in the validation errors.

Default settings:

- Resampling unit: contiguous validation-date blocks within one
  `(horizon, ticker)` series.
- Block length: 20 trading days.
- Bootstrap resamples: 1,000.
- Random seed: fixed and configurable.
- Metrics: MAE and RMSE.
- Pairwise comparisons:
  - `hist_gradient_boosting` versus `har_daily_weekly_monthly`;
  - `hist_gradient_boosting` versus `expanding_mean_abs_return`;
  - `har_daily_weekly_monthly` versus `expanding_mean_abs_return`.

Difference convention:

```text
difference = metric(model_a) - metric(model_b)
```

Negative values mean `model_a` had lower forecast error for that diagnostic.

Output rows should include:

| Column | Meaning |
| --- | --- |
| `horizon` | Forecast horizon |
| `ticker` | Market symbol |
| `metric` | `mae` or `rmse` |
| `model_a` | First model in the comparison |
| `model_b` | Second model in the comparison |
| `observed_difference` | Full validation-slice difference |
| `ci_lower` | Lower percentile interval bound |
| `ci_upper` | Upper percentile interval bound |
| `share_negative` | Share of resamples where difference is below zero |
| `block_length` | Bootstrap block length |
| `resamples` | Number of bootstrap resamples |

The default interval should be the 95% percentile interval. Reports may also
include 90% intervals later, but the first implementation should keep one clear
interval to avoid noisy interpretation.

## Interpretation Rules

Use these statements as the public boundary:

- Rolling wins are ranking diagnostics, not evidence of a universally best model.
- Bootstrap intervals describe forecast-error uncertainty under this validation
  design, not statistical proof of economic value.
- A pairwise result is stronger when the interval is narrow and mostly on one side
  of zero, but it still remains method-comparison evidence only.
- Mixed results across horizon, ticker, metric, or window should be reported as
  mixed results, not compressed into a single winner.

Avoid these claims:

- "ML beats HAR."
- "The model predicts volatility well enough to trade."
- "This result implies alpha, allocation value, or investment performance."
- "The best model is robust" unless the ranking and bootstrap diagnostics support
  that exact limited statement for a named horizon, ticker set, and metric.

## Acceptance Criteria For Milestone 3 Diagnostics

Implementation work should satisfy:

- forecast-error panels are generated reproducibly from tracked scripts;
- tests cover rolling-window construction and block-bootstrap resampling behavior;
- generated reports separate horizon, ticker, model, metric, and dates;
- Markdown reports include caveats and claim-boundary language;
- `uv run pytest`, `uv run ruff check .`, and report regeneration pass.
